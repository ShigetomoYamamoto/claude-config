# ADR-004: setup.sh はコピー型（シンボリックリンク非採用）

**ステータス**: accepted

**日付**: 2026-05-21

## コンテキスト

dotfiles の典型実装パターンには以下がある:

- (a) `cp` で配置するコピー型
- (b) `ln -s` でリンクするシンボリックリンク型
- (c) Stow や chezmoi など専用ツール型

どの方式を採用するか判断が必要。

## 検討した選択肢

1. **コピー型（cp）** — 配置先がリポジトリと独立。テンプレート展開・部分マージが可能
2. **シンボリックリンク型（ln -s）** — リポジトリの変更が即時反映。だが `__HOME__` 置換ができず、`mcp.json` の部分マージもできない
3. **chezmoi** — 高機能（テンプレート・暗号化対応）。だが外部ツール依存が増える
4. **Stow** — リンクベースで管理ファイルが整理される。だが部分マージができない

## 決定

**コピー型を採用する。**

冪等性を維持するため:

1. 各配置先を `rm -rf` してから `cp -r` する
2. `settings.json` はテンプレート → sed 置換
3. `mcp.json` は既存 JSON への追加マージ（不足分のみ）

## 結果

### Positive

- `~/.claude/` がリポジトリの clone 先と独立。リポジトリを別の場所に移動・削除しても Claude Code 設定は壊れない
- `__HOME__` のようなパス置換が可能（リンクではテンプレート展開ができない）
- MCP マージのような「部分的に既存を尊重する」処理が `mcp.json` で実現できる
- Stow 等の外部ツール依存がない（bash + python3 のみで動く）

### Negative

- 設定を変更したら毎回 `setup.sh` を再実行する必要がある（リンクなら自動反映）
- ユーザーが `~/.claude/agents/foo.md` を直接編集した場合、次回の `setup.sh` で消える（意図した挙動だが要注意）
