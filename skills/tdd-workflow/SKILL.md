# Test-Driven Development Workflow

This skill ensures all code development follows TDD principles with comprehensive test coverage.

## When to Activate

- Writing new features or functionality
- Fixing bugs or issues
- Refactoring existing code
- Adding API endpoints
- Creating new components or modules

## Core Principles

### 1. Tests BEFORE Code
ALWAYS write tests first, then implement code to make tests pass.

### 2. Coverage Requirements
- Minimum 80% coverage (unit + integration + E2E)
- All edge cases covered
- Error scenarios tested
- Boundary conditions verified

### 3. Test Types

#### Unit Tests
- Individual functions and utilities
- Component or class logic in isolation
- Pure functions, helpers, utilities

#### Integration Tests
- API endpoints
- Database operations
- Service interactions
- External API calls (with mocks for unit, real for integration)

#### E2E Tests (Playwright)
- Critical user flows
- Complete workflows end-to-end
- UI interactions

## TDD Workflow Steps

### Step 1: Write User Journeys
```
As a [role], I want to [action], so that [benefit]
```

### Step 2: Generate Test Cases

For each user journey, define:
- Happy path (success case)
- Validation errors (invalid input)
- Authorization errors (wrong role / unauthenticated)
- Edge cases (empty input, boundary values)
- Error scenarios (service unavailable, timeout)

### Step 3: Write Failing Tests (RED)

Write tests before implementation. Use the project's test framework (check `.claude/rules/testing.md` or `CLAUDE.md`).

```
describe "Feature name"
  test "happy path"
    # arrange: set up test data
    # act: call the function / endpoint
    # assert: verify expected outcome

  test "rejects invalid input"
    # arrange: invalid data
    # act
    # assert: error returned with appropriate status

  test "requires authentication"
    # arrange: no auth credentials
    # act
    # assert: 401 or equivalent rejected
```

Run tests — they **must FAIL** at this stage.

### Step 4: Implement (GREEN)

Write minimal code to make tests pass. Do not add functionality beyond what the tests require.

### Step 5: Run Tests Again

All tests must PASS. Fix the implementation if any fail — do not modify tests to pass.

### Step 6: Refactor (IMPROVE)

Improve code quality while keeping all tests green:
- Remove duplication
- Improve naming
- Optimize performance
- Enhance readability

### Step 7: Verify Coverage

Run the project's coverage tool (see `.claude/rules/testing.md` for the command).
Coverage must be ≥ 80%. If below, add tests for uncovered paths.

## Testing Patterns

### Unit Test: Arrange–Act–Assert

```
# Arrange: set up test data and mocked dependencies
subject = create_subject(dependency: mock)

# Act: invoke the unit under test
result = subject.do_something(input)

# Assert: verify the outcome
assert result == expected
assert mock.was_called_with(expected_args)
```

### Integration Test: API Endpoint

```
# Happy path
POST /api/resource  { valid_payload }
→ 201 Created, response contains new resource id

# Validation error
POST /api/resource  { invalid_payload }
→ 400 Bad Request, error message present

# Authentication required
POST /api/resource  (no auth header)
→ 401 Unauthorized
```

### E2E Test: Critical User Flow

```
navigate to start_page
fill in form fields with valid data
click submit button
expect success message to be visible
expect url to match success_path
```

## Test Isolation

- Each test sets up its own data — never depend on another test's side effects
- Roll back or clean up database state after each test
- Mock external services (email, payment, third-party APIs) in unit tests
- Do NOT mock the database in integration tests — use a dedicated test database

## Common Mistakes to Avoid

**Testing implementation details**
Test observable behavior (output, side effects, return values) — not internal state or private methods.

**Brittle selectors in E2E tests**
Use semantic selectors (role, label, data-testid) — not CSS class names that change with styling.

**Tests that depend on each other**
Each test must pass in isolation and in any order.

**Skipping the RED phase**
If a new test passes before implementation exists, it is not testing anything.

## Coverage Verification

Run the project's coverage command (see `.claude/rules/testing.md`).

Minimum thresholds:
- Branches: 80%
- Functions: 80%
- Lines: 80%

Identify uncovered paths from the report and add targeted tests.

## CI Integration

Tests and coverage run on every push. Check `.github/workflows/ci.yml` (generated by `/init-autonomous`) for the test job configuration.

---

**Remember**: Tests are not optional. They are the safety net that enables confident refactoring and production reliability.
